Java-详解ThreadLocal

发表于 2021-07-06 更新于 2022-07-13 分类于 技术分享
本文字数： 7.4k 阅读时长 ≈ 19 分钟

前言

ThreadLocal 类提供了线程内存储变量的能力，作用主要是做数据隔离，填充的数据只属于当前线程，对于别的线程而言是相对隔离的，在多线程环境下，可以防止自己的变量被其它线程篡改。

使用场景

数据隔离

ThreadLocal 主要用于各种开源框架中，比如：Spring的线程隔离、Mybatis的缓存机制（这些等我整明白了再补上）

实际项目很难用上 ThreadLocal，但有些场景确实需要使用它，比如：多线程使用 SimpleDateFormat

SimpleDateFormat 是一个线程不安全的类，我们使用它的 format() 和 parse() 方法来进行时间和字符串之间的转化。每次调用 format() 方法格式化时间时，都会重新设置内部的 Calendar 的对象，然后根据 calendar 对象中时间格式化字符串。调用 parse() 方法，也会修改的 Calendar 对象，然后根据 Calendar 创建日期对象。

因此，这个类在多线程的情况下使用可能会出现时间错误的问题。这个问题的解决方案就是使用 ThreadLocal 包装 SimpleDateFormat ，调用 ThreadLocal.withInitial() 方法为每个线程设置 SimpleDateFormat 对象副本

使用时通过 get() 方法获取对象副本，然后调用方法

传递上下文

除了数据隔离之外，ThreadLocal 也可以用来传递上下文。有时候我们项目中存在跨若干个方法调用，需要传递的对象（比如：用户身份）。但是为每个方法都添加上下文的参数非常麻烦，别更说中间调用了无法修改的第三方库。

在这种场景下可以使用 ThreadLocal 来设置上下文，需要使用到的地方 get() 一下就可以了。

存在的安全隐患

内存泄露

所谓内存泄漏就是程序中分配的内存由于某些原因无法释放（在 Java 中就是对象无法被回收），造成可用内存的减少。

为了便利大家理解，在讲 ThreadLocal 的内存泄漏问题前，我先讲一些前置知识

前置知识一、Java 的垃圾回收机制是如何判断对象是否可以回收的？

Hot Spot 是当下主流的 Java 虚拟机，它采用可达性分析法来进行对象判活。可达性分析法会从 GC ROOTS 出发向下搜素，搜素经过的路径被称为引用链，如果一个对象到 GC ROOTS 之间没有一条引用链，那么这个对象就是可以被回收的。

常见 GC ROOTS 包括：

Class–所有已加载的类

Thread – 活着的线程

Stack Local– - Java方法的局部变量或参数

JNI – navtive方法中引用的对象

前置知识二、Java 中的引用类型

Java 中一共有四种引用类型: 强引用、软引用、弱引用、虚引用。

强引用：Java中最常见的引用类型，把一个对象赋给一个引用变量，这个引用变量就是一个强引用；强引用的对象与 GC ROOTS 之间必然存在着一条引用链，因此不会被垃圾回收机制回收。它是造成内存泄漏的主要原因。

软引用：通过 SoftReference 创建，软引用的对象在内存不足时会被 GC 回收

弱引用：通过 Week Reference 创建，弱引用的对象在没有其他强引用的情况下，会在一下次 GC 时被回收

虚引用：通过 PhantomReference 创建，不会影响对象的生命周期，也无法通过虚引用来获取实例，为对象设置虚引用唯一的目的是为了让该对象被回收时返回一个系统通知

除了强引用外，其他三种引用都是通过类创建，这些类的父类 Reference 类有一个 referent 的字段，该字段引用的对象会被 GC 回收机制特殊处理。

ThreadLocalMap.Entry 继承了弱引用 WeekReference，在实例化时会调用父类 WeekReference 的构造方法传入 ThreadLocal 参数

而 WeekReference 又调用了父类 Reference 的构造方法，最终将 ThreadLocal 参数赋值给了 reference 字段。因此 Entry 中的 ThreadLocal 是一个弱引用。

综上所述，我们可以得出一个结论：ThreadLocalMap.Entry 与它的 Key（也就是 ThreadLocal）之间存在着弱引用关系，如果 ThreadLocal 对象没有其他外部的强引用，那么它会在下一次GC时被回收

下面通过一个案例来演示 ThreadLocal 的内存泄漏问题：我创建一个没有外部强引用的 ThreadLocal 对象并在当前线程中设置对象副本，接着打断点查看线程在 GC 前后 的 ThreadLocalMap

GC 前 ThreadLocalMap 条目中的 reference 为我们创建的 ThreadLocal对象，value 是我们设置的对象副本

GC 后弱引用的 ThreadLocal 对象被回收， reference 字段变为了空指针，而 value 却仍然存在。这是因为 value 对象与 GC ROOTS 之间存在着 thread->threadLocalMap->entry->value 这么一条强引用，所以它是不会被回收的，会导致内存泄漏。

对于内存泄漏问题，我们不必特别担心，因为在上面讲解的 ThreadLocalMap 源码中，我们可以看到很多的方法中都有清除过期条目的代码。比如：set() 、getEntry()方法在线性探测搜索条目时，会顺便清除过期节点；rehash() 方法会扫描全部的条目，清除过期节点。

虽然在源码中对内存泄漏做了很多工作，但上述的处理都具有一定的滞后性，我们平时使用完 ThreadLocal 后，最好是记得及时调用 remove() 方法清除条目。

脏数据

在线程池中使用 ThreadLocal 可能会读取到脏数据。这是因为线程池实现了线程的复用，某线程在执行某次任务时向 ThreadLocalMap 写入了数据，任务完成后此线程会存活一段时间。如果在这段时间内，该线程在执行其他任务时调用get() 方法就会读取到上次设置的数据。

我使用单例线程池来模拟同一个线程先后执行多个任务的场景：我每隔10秒钟向线程提交一个打印当前时间的任务。理论上每次打印的时间应该都差10s，但实际结果却是每次都打印了第一次时间。造成这个现象的原因就是线程重复时没有清除老数据。

这个问题的解决方法就是：主动调用 set() 或 remove() 方法清除或覆盖老数据。
