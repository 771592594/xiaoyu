Mybatis-缓存机制

发表于 2021-12-22 更新于 2023-02-05 分类于 技术分享
本文字数： 16k 阅读时长 ≈ 40 分钟

前言

在高并发的场景下，数据库的查询往往会成为一个性能的瓶颈。但实际上很多的查询是相同，在数据没有被修改的情况下返回的结果也是相同，如果我们对查询的结果做一个缓存，就可以减少数据库的压力。

Mybatis 是当前主流的持久层框架，它提供了两级的缓存机制，会将查询的数据自动保存到缓存中，后续相同的查询会直接从缓存中读取数据。下面让我们了解一下其缓存机制的使用方式和实现原理。

缓存实现

在 Mybatis 中，所有与缓存相关的类都存放在 cache 包下。 Cache 是缓存类的顶级接口，虽然它有很多的子类，但真正实现缓存行为的类只有 impl 目录下的 PerpetualCache；而 decorators 目录下的类都是装饰器（装饰器通过对实现类进行包装，实现对原有的方法进行增强），各个装饰器的作用详见《Mybatis-缓存机制中的装饰器》。

PerpetualCache

作为默认的唯一缓存实现类，PerpetualCache 的源码是非常简单的：它的字段只有一个 id 和一个名为 cache 哈希表。浏览它的方法，我们基本可以确定：这个缓存是一个 HashMap 实现的本地缓存。

一级缓存

Mybatis 的一级缓存是自动开启的，它是会话级别的，即在同一个 SqlSession 中，重复的查询会从缓存中读取结果。

实现原理

作用域

因为一级缓存的作用域是 SqlSession 级别的，因此它最适合存放的地方就是 SqlSession 的内部，因此我们可以尝试在 DefaultSqlSession 中找找看。它一共有如下五个字段，最后三个字段肯定是不可能存放缓存的，Configuration 保存的是全局配置也不适合存放，因此缓存最有可能存在于SQL的执行器 Executor 中。

BaseExecutor 是 Executor 的抽象类中，它定义了执行器基本的属性和增删改查的行为。在它的内部有一个缓存字段 localCache，在调用构造方法时 localCache 字段就会被实例化。

那么 Executor 又是何时被创建并被 DefaultSqlSessionFactory 引用的呢?

SqlSession 是由 SqlSessionFactory 创建的，那么在它的工厂方法 openSession() 中应该能够找到答案。

追踪工厂的默认实现类 DefaultSqlSessionFactory 的 openSession() 方法，该方法最终会调用openSessionFromConnection() 。在每次打开会话时，都会创建一个新的执行器给 DefaultSqlSession 使用。

当 DefaultSqlSession 关闭时，执行器会清空缓存。

二级缓存

Mybatis 的二级缓存需要手动配置，它是命名空间（namespace）级别的， 可以跨 SqlSession 使用。

使用方法

使用二级缓存需要两个步骤：

  1. 在 mybatis 全局配置中设置 mybatis.configuration.cache-enabled=true （默认配置）
  2. 在 xml 文件中添加 <cache> 标签，启用当前命名空间的二级缓存

为了测试二级缓存，我们需要将一级缓存案例中的 @Transactional 去掉，让两次查询在不同的会话中。

执行代码验证，可以看到虽然调用了2次方法，但只有第一次查询了数据库。日志还告诉我们该命名空间的缓存命中率为0.5，由此可见第二次查询使用了二级缓存。

实现原理

作用域

Mybatis 在创建 SqlSessionFactory 时，会将 xml 中定义的每个 select/insert/update/delete 标签都解析为一个 MappedStatement 对象，而二级缓存就存在于 MappedStatement 对象中。

在实现上，Mybatis 在解析 xml 文件时，会先创建一个 MapperBuilderAssistant 对象，然后调用它的 useNewCache() 方法，根据 <cache> 标签的属性创建缓存对象，然后将缓存对象记录到实例变量 currentCache 中。

在构造 MappedStatement 对象时，MapperBuilderAssistant 会将 currentCache 传入到构造的对象中。这样一来，同一命名空间下的标签会使用同一个二级缓存。

使用缓存

在全局配置中启用二级缓存后，Mybatis 会使用 CachingExecutor 装饰器来包装 Executor ，它的内部有一个事务缓存管理器 tcm。

在它的查询方法中，会使用 tcm 获取缓存的数据。

当时看这段源码的时候，我困惑了很久：这个 TransactionalCacheManager 类的 tcm 究竟是个什么东西呢?为什么不直接使用 MappedStatement 中的缓存呢？下面听我细细道来：

tcm 在调用 getObject() 和 putObject() 方法时，先会调用 getTransactionalCache 方法，使用 TransactionalCache 修饰器包装二级缓存，然后调用包装类的读写方法。

TransactionalCache 是二级缓存的事务缓冲区。所有查询出的数据会被写入到当前事务的缓冲区中，直到事务被提交的时候，才会真正地写入到二级缓存中。如果事务被回滚了，那么就会清除缓存。这样的做法，可以避免多个会话使用二级缓存时出现脏读问题。

缓存失效

先说结论：执行增删改操作会使该 namespace 的二级缓存失效

在一级缓存的失效中，我追踪源码得出一个结论： Mapper 对象的增删方法最终都是由 Executor 对象的 update() 方法完成。这个结论在二级缓存一样适用。

二级缓存的执行器外面包装了一层 CachingExecutor 的修饰器，因此会执行 CachingExecutor 的 update() 方法。这个方法会调用 flushCacheIfRequired() ，在必要的情况下清除 MappedStatement 对象中的二级缓存。

那么什么情况清除是必要的呢？

在默认情况，xml 文件中 insert/update/delete 标签对应的方法在执行时会自动清除缓存。我们可以通过修改标签的 flushCache 属性为 false 来禁用清除缓存。

自定义二级缓存

默认的二级缓存是基于本地内存的，在多实例的场景下使用会出现数据不一致的问题。因此，在这种场景下我们需要使用一个中间件（如：redis）作为缓存。

自定义缓存需要两个步骤：

  1. 创建自定义的缓存类实现Cache接口，且必须提供只接收 String 参数的构造方法。

  2. 在 xml 的 <cache> 标签中的 type 属性中传入自定义缓存类的全限定类名

为什么在第一步要求必须接收 String 参数?

原因是 CacheBuilder 使用 newBaseCacheInstance() 方法创建缓存时，会通过反射获取参数为 String 的构造方法，然后通过该构造方法接收缓存id，创建缓存对象。
