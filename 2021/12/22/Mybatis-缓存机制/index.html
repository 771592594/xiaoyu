Mybatis-缓存机制

发表于 2021-12-22 更新于 2022-01-17 分类于 技术分享
本文字数： 11k 阅读时长 ≈ 21 分钟

前言

在高并发的场景下，数据库的查询往往会成为一个性能的瓶颈。但实际上很多的查询是相同，在数据没有被修改的情况下返回的结果也是相同，如果我们对查询的结果做一个缓存，就可以减少数据的压力。

Mybatis 是当前主流的持久层框架，它提供了两级的缓存机制，会将查询的数据自动保存到缓存中，后续相同的查询会直接从缓存中读取数据。下面让我们了解一下其缓存机制的使用方式和实现原理。

缓存实现类

在 mybatis 中，所有与缓存相关的类都存放在 cache 包下。 Cache 是缓存类的顶级接口，虽然它有很多的子类，但真正实现缓存行为的类只有 impl 目录下的 PerpetualCache；而 decorators 目录下的类都是装饰器（装饰器通过对实现类进行包装，实现对原有的方法进行增强），这些装饰器用于实现二级缓存的特定功能（后面讲二级缓存的时候再提）。

作为默认的唯一缓存实现类，PerpetualCache 的源码是非常简单的：它的字段只有一个 id 和一个名为 cache 哈希表。浏览它的方法，我们基本可以确定：这个缓存是一个 HashMap 实现的本地缓存。

一级缓存

二级缓存

Mybatis 的二级缓存需要手动配置，它是命名空间（namespace）级别的， 可以跨 sqlSession 使用。

tcm 它只会在事务提交成功后，将数据真正地写入缓存。

使用方法

使用二级缓存需要两个步骤：

  1. 在 mybatis 全局配置中设置 mybatis.configuration.cache-enabled=true ，开启二级缓存
  2. 在 xml 文件中添加 <cache> 标签，启用当前命名空间的二级缓存

SynchronizedCache

SerializedCache

这个修饰器的作用是：

  • 在写入操作中，将插入的对象序列化为字节数组存储
  • 在读取操作中，将找出的字节数组反序列化为新的对象

当 <cache> 标签的 readOnly 属性为 false 时(默认配置)，缓存会被套上这个修饰器。这样每次读取的数据都是通过字节数组反序列化得到的新的对象，对该对象的修改不会影响缓存中的数据，更加安全。

由于序列化是通过 ObjectOutputStream 来实现的，因此数据模型类必须实现 Serializable 接口

当然，我们可以修改 readOnly 属性为 true 来禁用该修饰器，这样缓存就会存储对象本身，读写的效率更高，但对象的修改会直接影响到缓存中数据。只有当数据是只读的场景下，才适合这样配置。

LruCache

该修饰器的作用是设置缓存的淘汰策略为LRU（最近最少使用），该修饰器不需要手动配置，因为 <cache> 标签上的默认配置是 eviction="lru"。

实现原理：维护一个 LinkedHashMap 的子类对象，该对象重写了removeEldestEntry() 方法：当缓存中的数据量超出缓存指定的容量时，将最早插入的Key记录到实例变量 eldestKey 中。每次向缓存中插入数据后，若 eldestKey 非空则删除缓存中对应的数据。

那么 LinkedHashMap 是如何保证 removeEldestEntry() 方法的参数是最早插入的节点呢？

LinkedHashMap 内部维护着一条双向链表，每次通过 get(key) 方法时，都调用会 afterNodeAccess() 将该 key 对应的节点插入/移动到链表末尾。这样一来越近访问的的节点离尾节点越近，而头节点的 一定是最近最少访问的。

removeEldestEntry() 方法是在的插入节点后调用的，其参数是双向链表的头节点。该节点就是最近最少使用的节点。

FifoCache

该修饰器的作用是设置缓存的淘汰策略为FIFO（先入先出），需要在 <cache> 标签上配置 eviction="fifo"。

实现原理：维护一个保存Key的队列，每次向缓存中插入数据时，都先向队列中插入Key。当队列长度与缓存中键值对个数相同时，从队列中获取最早入队的Key，并在缓存中删除该数据。
