Java-缓冲流的实现与性能分析

发表于 2021-08-17 分类于 技术分享

前言

在使用IO流时，我经常会习惯性地包装一层缓存流，因为上网的案例都是这么写的。例如，向文件写入一串文本时，我会创建一个 FileOutputStream，然后再用 BufferedOutputStream 进行包装。

对于缓存流，很多人只知道它内部提供了一个缓冲区，能够提高程序的运行效率。那么缓冲流的原理究竟是什么呢？使用缓存流真的比直接使用节点流效率更高吗？下面让我通过源码和实际的案例来解答这两个问题。

装饰器模式

在讲下面源码之前，先介绍一下装饰器模式。这种模式创建了一个装饰类，用来包装原有的类，在原有类方法的基础上提供了额外的功能。UML 图如下所示：

使用装饰器模式分为四步：

第一步，创建 Component 类充当抽象角色

第二步，为 Component 提供一个基本的实现类

第三步，创建修饰类引用并继承 Component 类，重写父类方法实现增强

第四步，创建包装类 ComponentWrapper 并引用 Component 的实例。这样我们就可以对原有的方法进行增强

在 Java 的 IO 流中使用了大量的装饰器模式：InputStream 是所有输入流的基类，提供了基本的方法。而 FilterInputStream 及其子类是节点流的包装类，它的内部封装了节点流，通过重写方法对节点流的功能进行增强。

流按照角色可以分为：节点流和处理流

节点流：直接向节点读写数据的流。例如：读字节数组的 ByteArrayInputStream、读文件的 FileInputStream；

处理流：对节点流进行包装以提供额外功能的流，所有的处理流都是 FilterInputStream 的子类。例如：基于缓冲区进行读取的 BufferedInputStream。

缓冲流的原理和性能分析

IO流可以按照流向分为 输入流 和 输出流，我将分别以 BufferedInputStream 和 BufferedOutputStream 的为例进行分析。

BufferedInputStream

BufferedInputStream 内部维护一个字节数组作为缓存区，并且记录缓冲区的大小和下次读取的位置。创建时包装节点流，并初始化缓冲区大小。

BufferedInputStream 通过 getBufIfOpen() 方法来获取缓存区，通过 getInIfOpen() 方法获取被包装的节点流。

它的 read(byte b[], int off, int len) 方法可以将数据读取到参数 byte[] b 中，里面调用 read1() 方法来读取数据，具体逻辑见注释。

fill() 最终调用节点流的 native int readBytes() 一次加载 buffer.length 个（默认8K）字节到缓冲区。

综上所述，缓冲流的原理就是每当缓冲区为空时，调用本地方法读取大量的数据填充缓冲区，之后直接读取缓冲区中的数据。而节点流每次读取数据都是调用本地方法的，本地方法相比Java方法，效率较低。因此通过设置缓冲区，减少本地方法的调用，可以提高程序的效率。

以读取一个1MB的文件为例对缓冲流和节点流的性能进行对比，缓冲区大小为默认的8192，每次读取1024个字节。缓冲流只需要调用2次本地方法填充缓冲区，其余读取都是基于缓冲区来完成的，而节点流需要调用10次本地方法来获取数据。
