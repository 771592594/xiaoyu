Java-线程池详解

发表于 2021-06-12 更新于 2021-07-25 分类于 技术分享

为什么使用线程池

在开发过程中，我们经常会创建多个线程来处理多个任务，提高程序的运行效率。

这种写法虽然简单，单是创建的线程存在着以下问题：

  1. 系统开销大。通过这种方式创建的线程是 一次性用品，在执行完成任务后就会被销毁。但是对操作系统来说，创建线程是一个耗时的操作。如果每次我们需要使用线程时都让操作系统创建线程，系统开销会比较大。
  2. 存在安全隐患。每个线程都需要分配一定的内存，默认一个线程的线程栈大小是1M。若不对线程的创建加已控制，无限地创建线程，系统会因内存溢出而崩溃。
  3. 不便于统一管理。通过这种方式创建的线程是一个个零零散散的对象，我们很难对其进行统一的管理和配置。

为了解决上述的问题，我们一般会采用线程池来管理线程

使用线程池的好处有：

  1. 降低系统开销，提高响应速度
    线程池能够重复利用已创建的线程来执行任务，任务可以不需要等到线程创建就能立即执行，减免了线程创建和销毁造成的开销，提高了系统的响应速度。

  2. 对线程进行统一管理，提高系统的稳定性
    使用线程池可以对线程进行统一分配、 管理和监控，具体表现为：线程池可以限制创建的线程个数；自动销毁空闲线程；通过线程工厂对创建的线程进行统一的配置。

线程池的创建方式

Java提供了 ThreadPoolExecutor 类创建线程池，它的构造方法一共有7个参数

corePoolSize: 核心线程数，线程池的在空闲情况下所能够保持的线程数。

maximumPoolSize: 线程池所能保存的最大线程数量；如果使用了无界的任务队列，则该参数无效。

keepAliveTime: 线程的生存时间

当线程数大于核心线程数时，若线程在一定时间内处于没有执行任务，线程池就会自动销毁该线程。线程池不会销毁所有线程，最多保存corePoolSize个线程。

TimeUnit: 线程生存时间的单位

BlockingQueue: 任务队列

用于保存待执行任务的阻塞队列（线程安全的队列）。可选队列类型如下：

  • ArrayBlockingQueue：基于数组的有界阻塞队列，按先进先出原则对元素进行排序。

  • LinkedBlockingQueue：基于链表的阻塞队列，如果创建时设置了size参数就是有界队列，否则是无界队列。

  • SynchronousQueue：不存储元素的阻塞队列。每个插入操作必须等到另一个线程调用移除操作，否则插入操作一直处于阻塞状态，吞吐量较高。

  • PriorityBlockingQueue: 具有优先级的无限阻塞队列。

ThreadFactory: 线程工厂

用于创建线程的工厂，可以在创建方法可以自定义线程的基本属性（如：名称、优先级、设置守护线程）

RejectedExecutionHandler: 饱和策略

当阻塞队列和线程池都满了,说明线程池处于饱和状态，那么必须采取一种策略处理提交的新任务。在JDK中提供了以下4种策略：

  • AbortPolicy：直接抛出异常（默认）
  • DiscardPolicy：直接丢弃当前任务
  • DiscardOldestPolicy：丢弃队列里最近的任务，并执行当前任务
  • CallerRunsPolicy：只用调用者所在线程来运行任务

线程池工作流程

线程池是一个经典的生产者-消费者的模型，它的工作流程如下所示：

  1. 使用者向线程池提交了一个任务

  2. 如果当前运行的线程数少于corePoolSize，则创建新线程来执行任务

  3. 如果运行的线程数大于等于corePoolSize，则将任务加入阻塞队列

  4. 如果阻塞队列已满，则创建新的线程来处理任务

  5. 如果创建新线程将使当前运行的线程超出最大线程数，则采取饱和策略处理

上述步骤的总体设计思路是为了在提交任务后，尽可能地避免创建线程，减少系统的开销。在线程池完成预热之后（当前线程数大于等于核心线程数），几乎所有的新提交的任务都会被存到工作队列中。

线程池的生命周期

状态 说明
running 线程池处于运行状态，能够接收新任务，同时处理已提交的任务
shutdown 线程池调用 stop() 方法进入该状态。在该状态下不再接收新任务，但会处理已提交的任务，线程在执行完成队列中的所有任务后结束
stop 线程池调用 stop() 方法进入该状态。在该状态下不再接收新任务，也不再处理已提交的任务，尝试中断现有线程，返回等待队列中执行任务的列表
tidying 当所有任务都已终止，且有效线程数为0时，进入tidying状态，然后调用钩子函数 terminated()
terminated 线程池彻底终止

线程池配置

要想合理地配置线程池，就必须首先分析任务特性，性质不同的任务用不同规模的线程池分开处理。可以从以下几个角度来分析：

  • 任务的性质: CPU密集型任务、I0密集型任务
  • 任务的优先级
  • 任务的执行时间
  • 任务的资源依赖性

CPU密集型与IO密集型

CPU密集型任务应配置尽可能小的线程，如配置N+1个线程的线程池（通过Runtime.getRuntime().availableProcessors() 方法获得设备的CPU数目）

执行IO密集型任务的线程大多时间是被IO阻塞的，实际上没执行任务，所以应配置尽可能多的线程，如：2 * N

任务优先级

优先级不同的任务可以使用优先级队列PriorityBlockingQueue来处理。它可以让优先级高的任务先执行。注意如果一直有优先级高的任务提交到队列里,那么优先级低的任务可能永远不能执行。

执行时间

执行时间不同的任务可以交给不同规模的线程池来处理，或者可以使用优先级队列，让执行时间短的任务先执行。

任务的资源依赖性

依赖其他资源的任务需要根据实际情况进行配置线程池。比如：依赖数据库连接池的任务，在线程提交 SQL 后需要等待数据库返回结果，等待的时间越长，则CPU空闲时间就越长，那么线程数应该设置得越大,这样才能更好地利用CPU。

线程池实现原理

线程复用

线程池有一个内部类叫 Worker，它实现了 Runnable 接口，它有2个重要的内部变量 thread (创建的线程) 和 firstTask (线程的首任务)

Worker 的构造方法接收一个 Runnable 参数，将其赋值给 firstTask，还会使用线程池的线程工厂的 newThread(Runnable task) 方法创建线程，将自身作为参数传入。这样一来，当创建的线程执行时，就会执行 Worker对象 的 run() 方法，实际就是执行 runWorker() 方法。

在 runWorker() 中，当 Worker 对象的 firstTask 没有执行时，会执行它的 firstTask；若已执行，则会从阻塞队列中获取任务执行，这个过程会持续循环直到线程停止为止。至此就实现了线程的复用。

自动销毁

当线程数大于核心线程数时，若线程在一定时间内处于没有执行任务，线程池就会自动销毁该线程。在默认未开启核心线程过期的情况下，线程池不会销毁所有线程，最多保存corePoolSize个线程。

在线程复用的实现中，线程通过阻塞队列的 take() 或 poll(long timeout, TimeUnit unit) 方法会不断地获取任务执行，具体分两种情况：

如果当前线程数小于总线程数，线程会使用 take() 方法从工作队列中获取任务，如果队列为空，线程会进入WAITING状态，直接队列非空才会被唤醒，然后重试直到获取任务成功。在这种情况下，线程不会被销毁。

如果当前线程数超过总线程数，线程会使用 poll(long timeout, TimeUnit unit) 方法从工作队列中获取。如果队列为空，线程会进入 TIMED_WAITING（超时等待）状态，在休眠一段时间后醒来后，如果仍然无法获取到任务，该线程就会被销毁。

Executors

Executors 是 ThreadPoolExecutor的工厂类，它的提供了四种快速线程池的静态方法，本质是对ThreadPoolExecutor的构造方法预设参数。

这个类提供的线程池方案都存在着一定的安全隐患，因此在阿里巴巴开发规范中被禁止使用。我们平时写demo的时候可以用来节约时间，但生产中建议不要使用。

下面简单介绍一下它提供的四种线程池：

一、newSingleThreadExecutor

它创建的线程池的特点是：

  • 核心线程池数和总线程池数都为1
  • 线程没有过期时间，会一直存在
  • 采用基于链表的无界阻塞队列

二、newFixedThreadPool

它创建线程池的特点：

  • 核心线程数与总线程数为都为N
  • 线程没有过期时间，会一直存在
  • 采用基于链表的无界阻塞队列

三、newCachedThreadPool

newCachedThreadPool 创建线程池的特点：

  • 核心线程数为0
  • 总线程数为 Integer.MAX_VALUE
  • 线程的过期时间为60秒
  • 使用同步队列（特点是：容量为1，在没有元素中，可以存入；有元素时，不能直接取出，只有存入新元素的时候才能取出队列中的元素）

四、newScheduledThreadPool

newScheduledThreadPool 创建线程池的特点：

  • 核心线程数可以自定义
  • 最大线程数为 Integer.MAX_VALUE
  • 线程没有过期时间，会一直存在
  • 采用专门的延迟队列，可以执行延迟任务

看完上面到这里，你可能会有点疑惑，为什么它会被阿里开发规范禁止使用呢？

  1. 有些方法创建的线程池使用无界队列，如果队列中大量任务堆积，存在内存溢出的风险
  2. 有些方法创建的线程池将最大线程数设置为 Integer.MAX_VALUE，相当于允许无节制的创建线程，也同样会耗尽系统资源。
