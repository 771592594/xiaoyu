为什么使用线程池

线程是一种贵重的系统资源，每次创建和销毁都需要较大的开销，如果不对线程进行统一的管理，程序的效率和稳定性都会由很大的影响。

下面看一个案例，在这个案例中用户每次请求都会创建一个线程来处理任务。

这段代码非常简单，但也存在着以下的问题：

  1. 线程作为操作系统中重要的资源，创建的数量是非常有限的，并且创建和销毁都需要巨大的系统开销。如果每次用户请求都创建一个新的线程来处理，那么在高并发的情况下，系统中存在大量的线程，任务切换会花费更多的时间，导致程序的 运行效率下降。

  2. 线程本身也需要占用内存空间，无节制地创建会导致 内存溢出

如果我们能限制线程创建的数量，对创建的线程进行复用，就能解决上面的问题。为此，就有人提出了池化思想：将有限的资源放在一个资源池中，需要资源的时从池子中拿，用完之后放回池子里，后面可以接着用。我们编程时所用的连接池、常量池，以及今天要讲的线程池都是这一思想的具体实现。

使用线程池的主要作用有：

  1. 实现线程复用，降低系统开销

线程池能够重复利用已创建的线程来执行任务，任务可以不需要等到线程创建就能立即执行，减免了线程创建和销毁造成的开销，提高了系统的响应速度。

  1. 对线程进行统一管理，提高系统的稳定性

使用线程池可以对线程进行统一分配、 管理和监控，具体表现为：线程池可以限制创建的线程个数；自动销毁空闲线程；通过线程工厂对创建的线程进行统一的配置。

线程池的创建方式

创建线程池有常用的方式有两种：

一、ThreadPoolExecutor 的构造方法创建线程池

二、使用工厂类 Executors 创建线程池，由于一些问题在生产中不推荐使用，具体见后文

ThreadPoolExecutor

ThreadPoolExecutor 的构造方法一共有7个参数，学习时先大概了解一下线程池的参数作用，然后结合线程池的运行原理进行理解。

corePoolSize: 核心线程数，线程池的在空闲情况下所能够保持的线程数。

maximumPoolSize: 线程池所能保存的最大线程数量，如果使用了无界的任务队列，则该参数无效。

keepAliveTime: 线程的生存时间，当线程数大于核心线程数时，若线程在一定时间内处于没有执行任务，线程池就会自动销毁该线程。线程池不会销毁所有线程，最多保存corePoolSize个线程。

TimeUnit: 线程生存时间的单位

BlockingQueue: 任务队列，用于保存待执行任务的阻塞队列。常用的类型如下：

  • ArrayBlockingQueue：基于数组的有界阻塞队列，按先进先出原则对元素进行排序。

  • LinkedBlockingQueue：基于链表的阻塞队列，如果创建时设置了size 参数就是有界队列，否则是无界队列。

  • SynchronousQueue：不存储元素的阻塞队列。每个插入操作必须等到另一个线程调用移除操作，否则插入操作一直处于阻塞状态，吞吐量较高。

  • PriorityBlockingQueue: 具有优先级的无限阻塞队列。

ThreadFactory: 用于创建线程的工厂，可以在创建方法可以自定义线程的基本属性（如：名称、优先级等）

RejectedExecutionHandler: 饱和策略，当线程池无法接收新任务时，所采取的策略。在JDK中提供了以下4种策略：

  • AbortPolicy：直接抛出异常（默认）
  • DiscardPolicy：直接丢弃当前任务
  • DiscardOldestPolicy：丢弃队列里最近的任务，并执行当前任务
  • CallerRunsPolicy：只用调用者所在线程来运行任务

线程池工作原理

线程池是一个经典的生产者-消费者的模型，它的工作原理如下所示：

  1. 使用者向线程池提交了一个任务

  2. 如果当前运行的线程数少于corePoolSize，则创建新线程来执行任务

  3. 如果运行的线程数大于等于corePoolSize，则将任务加入阻塞队列

  4. 如果阻塞队列已满，则创建新的线程来处理任务

  5. 如果创建新线程将使当前运行的线程超出最大线程数，则采取饱和策略处理

上述步骤的总体设计思路是为了在提交任务后，尽可能地避免创建线程，减少系统的开销。在线程池完成预热之后（当前线程数大于等于核心线程数），几乎所有的新提交的任务都会被存到工作队列中。

线程池的生命周期

状态 说明
running 线程池处于运行状态，能够接收新任务，同时处理已提交的任务
shutdown 线程池调用 stop() 方法进入该状态。在该状态下不再接收新任务，但会处理已提交的任务，线程在执行完成队列中的所有任务后结束
stop 线程池调用 stop() 方法进入该状态。在该状态下不再接收新任务，也不再处理已提交的任务，尝试中断现有线程，返回等待队列中执行任务的列表
tidying 当所有任务都已终止，且有效线程数为0时，进入tidying状态，然后调用钩子函数 terminated()
terminated 线程池彻底终止

线程池实现原理

线程复用

线程池有一个内部类叫 Worker，它实现了 Runnable 接口，它有2个重要的内部变量 thread (创建的线程) 和 firstTask (线程的首任务)

Worker 的构造方法接收一个 Runnable 参数，将其赋值给 firstTask，还会使用线程池的线程工厂的 newThread(Runnable task) 方法创建线程，将自身作为参数传入。这样一来，当创建的线程执行时，就会执行 Worker对象 的 run() 方法，实际就是执行 runWorker() 方法。

在 runWorker() 中，当 Worker 对象的 firstTask 没有执行时，会执行它的 firstTask；若已执行，则会从阻塞队列中获取任务执行，这个过程会持续循环直到线程停止为止。至此就实现了线程的复用。

自动销毁

在线程复用的实现中，线程通过不断地从工作队列中获取任务，如果没有拿到任务就会被销毁

获取任务地过程，又具体分为两种情况：

如果当前线程数小于总线程数，线程会使用 take() 方法从工作队列中获取任务，如果队列为空，线程会进入WAITING状态，直接队列非空才会被唤醒，然后重试直到获取任务成功。

如果当前线程数超过总线程数，线程会使用 poll(long timeout, TimeUnit unit) 方法从工作队列中获取。如果队列为空，线程会进入 TIMED_WAITING（超时等待）状态，在休眠一段时间后醒来后，如果仍然无法获取到任务，则返回null

线程池配置

线程池的意义在于复用线程，但并不是说整个程序都使用一个线程池。实际开发中，我们要根据任务的具体情况来创建和配置线程池。具体可以参考的规则：

  1. 创建线程池时，要通过线程工厂参数为线程设置有意义的名称，方便排查问题时进行定位

  2. 线程池中的线程数不是越多越好，过多的线程会增加线程切换的开销

  3. 性质不同的任务用不同的线程池进行处理。CPU密集型任务应配置尽可能小的线程，如配置N+1个线程（N为CPU核心数）；IO密集型任务的线程大多时间是被IO阻塞的，应配置尽可能多的线程，如：2 * N

  4. 线程池应根据线上情况进行配置，一开始先设置足够大的值保证正常运行，后面根据实际情况进行调整。为此我们需要对线程池进行监控。简单的监控实现就是开启一个线程，不断打印线程池的信息：

  5. 如果存在任务大量堆积在队列中的情况，可以考虑魔改线程池，在线程数未达到总线程数时，优先创建新线程来处理任务；线程数达到最大后，将任务存放到队列中。实现方式是自定义工作队列，修改 offer() 方法。

不推荐的Executors

Executors 是 ThreadPoolExecutor的工厂类，它的提供了快速创建线程池的静态方法，本质是对ThreadPoolExecutor的构造方法预设参数。

这个类提供的线程池方案都存在着一定的安全隐患，因此在阿里巴巴开发规范中被禁止使用。我们平时写demo的时候可以用来节约时间，但生产中千万不要使用。

下面简单介绍一下它提供的其中两种线程池方案，以及各自存在的风险：

一、newFixedThreadPool

它创建线程池的特点：

  • 核心线程数与总线程数为都为N
  • 线程没有过期时间
  • 采用基于链表的无界阻塞队列（风险：任务堆积时，存在内存溢出风险）

二、newCachedThreadPool

newCachedThreadPool 创建线程池的特点：

  • 核心线程数为0
  • 总线程数为 Integer.MAX_VALUE
  • 线程的过期时间为60秒
  • 使用不存放元素的同步队列（风险：SynchronousQueue 是一个没有存储空间的阻塞队列，这意味着只要有请求到来，就必须找到一条工作线程来处理，如果当前没有空闲的线程就再创建一条新的，总线程数最多可达 Integer.MAX_VALUE，存在内存溢出风险）
