Java-SPI机制的理解与应用

发表于 2021-06-29 更新于 2021-08-09 分类于 技术分享

前言

SPI (Service Provider Interface) 是一种服务发现机制。它会在运行时从 ClassPath 路径下的 META-INF/services 目录查找服务配置文件，并根据文件中配置的类名加载类。这一机制实现了组件之间的解耦，为系统的扩展提供了可能，在 JDBC、SLF4J、Dubbo 都有应用。

在第二节 如何使用 SPI 中，我是通过遍历迭代器来加载类并创建对象。在遍历的过程中，迭代器究竟做了什么呢？让我们接着向下看：

懒迭代器 LazyIterator 是 ServiceLoader 的一个内部类，我们通过迭代器提供的 hasNext() 和 next() 接口对元素进行遍历，它们底层又分别调用了 hasNextService() 和 nextService() 方法。

hasNextService() 方法会根据服务（接口）的全限定类名从 META-INFO/Service 目录中加载配置，然后转化为一个枚举对象 configs，并读取下一条配置（每条配置都是服务的实现类名）。

nextService() 方法会调用 Class.forName() 加载当前遍历到的服务并将其实例化后返回。

总结

SPI 机制使用需要以下条件：

  1. 项目中存在服务接口
  2. 通过 jar 包的方式，引入接口的实现类和配置文件
  3. 配置文件存放在 ClassPath 的 META-INFO/Service 目录下，并且文件名与服务接口的全限定类名相同
  4. 在配置文件中添加需要加载的实现类的全限定类名
  5. 在代码中的某处调用了 ServiceLoader.load() 动态地加载类

SPI 机制本质是通过面向接口编程的方式实现组件之间的解耦，我们可以根据自己的需要引入不同的接口实现，而不需要修改原有的代码，这样就提高项目的可扩展性。
