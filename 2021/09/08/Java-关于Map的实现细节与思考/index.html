Java-关于Map的实现细节与思考

发表于 2021-09-08 更新于 2021-09-20 分类于 技术分享

HashMap的容量永远是二的幂

一、HashMap 内部维护一个哈希表 Node<K,V>[] table， 哈希表的容量永远是二的幂。这是基于两个事实得出的结论：

  1. 哈希表的初始容量永远是二的幂

HashMap 会在第一次插入元素时，会对 table 其进行初始化，初始化的容量通过 tableSizeFor() 方法计算得到 。这个方法会返回一个 不小于当前容量且是二的幂的数。

  1. 哈希表每次扩容，容量都会变为原来的两倍

resize() 方法会创建容量为原哈希表两倍的新表，由于原哈希表的初始容量永远是二的幂，所以新表容量也是二的幂。

二、为什么 HashMap 要让哈希表的容量一直是二的幂呢？

这个跟 HashMap 计算元素的位置的公式如下，这个公式通过 与运算 快速地计算出查询/插入元素的位置

$$
i\ =\ (capacity- 1) \ and \ hash
$$
为了保证散列结果的均匀分布，哈希表的容量必须为2的幂，因为只有 2的幂 - 1 才能得到全1的二进制数，只有全1的二进制数和随机的哈希值做与运算才能保证结果是 [0, capacity - 1] 中的任何数。

假设哈希表的长度不是二的幂，比如10，它的二进制是1010，1010与任何数做与运算的结果都是 ?0?0，能够插入的位置只有以下4种可能：0(0000)、2(0010)、8(1000)、10(1010)。

这样哈希冲突的概率非常的高，会形成长链表，严重影响查询效率。

并发容器的键值不能为null

如果允许并发Map的键值为null，那么在多线程情况下，我们使用时 get() 方法得到一个 null 时，该如何判断是不存在该key 还是 存在值为null的映射呢？

在单线程使用时，我们可以通过 containsKey() 来判断；但是在多线程情况下，其他线程随时可能增删 KEY，该方法只能反映 key 在调用的时间点是否存在。因此为了避免歧义，设计者直接禁止 KEY 和 VALUE 为 null。

举个例子：我们想要执行这样的逻辑：如果从Map中根据 key 拿到 null 值并且这个key 存在，则执行业务逻辑。

如果允许键值为null，那么在多线程情况下这段代码并不能像我们想的那样执行，因为存在着这样一个可能性：

线程一调用 get() 方法，此时不存在 key，所以得到了 null，然后时间片结束切换到线程二。

线程二执行其他方法时，向Map中插入了这个key，然后切换回线程一。

线程一此时执行 if 语句，发现 key 是存在的，就执行了业务逻辑，这与我们期望的并不一致的。
