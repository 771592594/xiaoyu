快速入门

项目搭建

要在 Maven 项目使用 Hibernate Validator，你需要将如下依赖添加到 pom.xml 文件中

Hibernate Validator 需要一个 Jakarta Expression Language 规范的实现，用于执行动态表达式（EL）获取约束违反信息。当你的应用运行在一个 Jakarta EE 容器（例如：Tomcat, JBoss）中时，EL 实现已经由容器提供。然而，当你的应用运行在 Java EE 环境时，你必须在 pom.xml 添加一个EL实现的依赖。

使用约束

除了上面使用的注解，你还可以在 jakarta.validation.constraints 包下找到其他约束注解

校验约束

使用注解声明约束后，还需要一个校验器 Validator 实例来执行约束的验证，让我们看一个 Car 类的单元测试。

在 setUpValidator() 方法中，一个 Validator 对象从 ValidatorFactory 中取出。Validator 实例是线程安全的，可以重复使用。因此可以安全地被存储到静态字段中，用于不同对象实例的校验。

validate() 方法返回一个 ConstraintViolation 实例，你可以遍历它来查看发生的校验错误。前3个测试用例展示了我们期望的约束违例：

  • 在manufacturerIsNull()方法里， manufacturer 字段上的 @NotNull 约束是违反的
  • 在licensePlateTooShort() 方法里，licensePlate 字段上的 @Size 约束是违反的
  • 在 seatCountTooLow()方法里， seatCount 字段上的 @min 约束是违反的、

如果对象验证成功，validate() 方法返回一个空集合，你可以在 carIsValid() 用例中看到。

对象约束的声明与校验

声明对象约束

约束在 Jakarta Bean Validation 中是通过注解来表达的，对象约束分别4个类型：

  • 字段约束
  • 属性约束（属性：私有成员变量及其getter/setter）
  • 容器元素约束
  • 类约束

下面将分别介绍每种约束的声明方法

字段约束

当使用字段级别约束时，验证引擎会直接访问实例对象进行验证，而不会执行get方法。

约束可以被应用于任何访问级别的字段上（例如：public, private），但是不能用于静态字段。

属性约束

属性指的是私有成员变量及其getter/setter

当使用属性级别约束时，验证引擎会通过属性方法来访问被验证的值

容器元素约束

容器元素约束指的是直接在泛型的类型参数上设置的约束

Hibernate 支持以下标准Java容器的容器元素约束 ：

  • Iterable 接口实现
  • Map 接口实现
  • Optional

自定义的容器类型可以通过提供 ValueExtractor 接口实现来支持（TODO）

类级约束

约束继承

类实现接口 或者 继承另一个类时，字段和方法中的注解也会被继承下来，因此接口和父类中的约束会被继承到子类中

级联校验

Jakarta Bean Validation API 不仅仅支持单个对象实例的校验，还能完成整个对象图的校验。

要实现整个对象图的校验，只需要在另一个对象的字段引用上加上@Valid 注解即可。

获取校验器

通过Validation 静态工厂方法可以轻松地创建一个校验器

校验器API

validate

最常用的校验API，能够对传入object对象的所有约束进行校验，并返回校验结果。

validateProperty

对object的propertyName属性进行校验，并返回校验结果

validateValue

使用beanType的propertyName属性的约束规则对值进行校验，并返回校验结果

分组校验

之前章节讨论的 Validator 和 ExecutableValidator 所有的校验方法中都接收一个可变长的 groups 参数。至今为止我们都是忽视它的，但在本章将介绍它的作用。

请求分组

分组允许我们限制校验时使用的约束集。目标组作为var-arg参数传递给适当的验证方法。

让我们看一个例子，Person 类的 name 字段上有个 @NotNull 约束。这个约束上没有指定任何分组，它会被默认设为 jakarta.validation.groups.Default 分组。

Driver 类继承了 Person ，添加了age 和 hasDrivingLicense 属性。约束条件是：司机必须要年满18岁(@Min(18))且拥有驾照(@AssertTrue)。定义在字段上的约束条件都属于 DriverChecks 分组，而 DriverChecks 只是一个用于打标签的简单接口。

最后， Car 类有一些约束，它们是默认分组的一部分，以及 CarChecks 分组中 passedVehicleInspection 属性的@AssertTrue，该属性指示汽车是否通过了路况测试。

在这个例子中总共使用了三个不同的组:

  • Person.name, Car.manufacturer, Car.licensePlate 以及Car.seatCount 字段的约束都属于 Default 分组
  • Driver.age 和 Driver.hasDrivingLicense 字段的约束都属于 DriverChecks 分组
  • Car.passedVehicleInspection 字段的约束属于 CarChecks 分组

定义分组序列

默认情况下，不管属于哪个分组，约束都会被无序地执行。然而在一些场景下，我们需要控制约束的执行顺序。

为了实现有序校验，只需要定义一个接口并用 @GroupSequence 修饰它，然后在分组序列中定义分组的执行顺序。注意：如果一个分组序列中的一个约束校验失败了，序列中后续的分组都不会被校验。

重定义默认分组

@GroupSequence

除了定义分组序列之外，@GroupSequence 注解还可以用来重定义类的默认分组。只需要将 @GroupSequence 注解添加到指定的类上，并且指定取代 Default 的分组序列。

通过这个定义，你可以执行通过默认分组来执行属于 RentalChecks, CarChecks 和 RentalCar 分组的约束校验。

@GroupSequenceProvider

除了通过 @GroupSequence 静态地重定义分组序列，Hibernate Validator 也提供了基于对象状态动态重定义默认分组的SPI机制
